The mod uses 7 ASM ClassVisitor adapters registered via CoreModTransformer, plus reflection in ShipIntermediary.getEntityDistanceSq. Each adapter hooks one or two methods. All will be replaced with Mixin @Inject (or @Redirect) classes that call the existing ShipIntermediary/ShipIntermediaryClient static methods unchanged.
Steps
Create MixinWorld targeting World — replace WorldAdapter's two injections: @Inject at HEAD of checkBlockCollision (return early if ship collision), and @Inject at RETURN of getEntitiesWithinAABBExcludingEntity (append ships to result list).
Create MixinEntity targeting Entity — replace EntityMoveAdapter and EntityDistanceAdapter: @Inject at HEAD of moveEntity (call preEntityMove), @Inject at RETURN of moveEntity (call postEntityMove), and @Inject at HEAD of getDistanceSqToEntity (call intermediary, cancel with result if ≥ 0).
Create MixinEntityLivingBase targeting EntityLivingBase — replace EntityLadderAdapter: @Inject at HEAD of isOnLadder, cancel with true if ShipIntermediary.isEntityOnShipLadder returns true.
Create MixinEntityPlayer (extend the existing one) — replace EntityPlayerAdapter: @Inject at HEAD of wakeUpPlayer (call onPlayerWakeUp), and @Inject at HEAD of isInBed (cancel with true if ShipIntermediary.isPlayerInBed returns true).
Create MixinServerConfigurationManager targeting ServerConfigurationManager — replace ServerConfigurationManagerAdapter: @Redirect on the ChunkProviderServer.loadChunk call inside respawnPlayer to inject onPlayerRespawn just before the original call.
Create MixinEntityRenderer (client-only) targeting EntityRenderer — replace EntityRendererAdapter: @Inject at RETURN of getMouseOver to call ShipIntermediaryClient.onFoundHit.
Create MixinTileEntityInventory + MixinContainer — replace TileEntityInventoryAdapter's two injections: @Redirect on EntityPlayer.getDistanceSq inside isUseableByPlayer/canInteractWith (route to ShipIntermediary.getEntityDistanceSq), and @Redirect on the worldObj field write in Container <init> (route to ShipIntermediary.translateWorld). Note: because these target any class implementing the interface, these may need @Mixin on multiple concrete classes or require a broad approach — worth discussing.
Replace ShipIntermediary.getEntityDistanceSq reflection — the overload taking a Container uses reflection to read worldObj, posX, posY, posZ. Create IMixinContainerWorkbench, IMixinContainerEnchantment, and IMixinContainerRepair @Accessor interfaces, or create a shared IMixinContainerWithWorld interface and use @Accessor to expose the fields directly, eliminating the getField() reflection.
Register all new Mixins in the Mixins.java enum, then delete all 7 adapter files, CoreModTransformer.java, and remove the getASMTransformerClass entry from CoreModPlugin.java.
Further Considerations
TileEntityInventoryAdapter targets all implementors — it hooks any class whose <init> takes (InventoryPlayer, World, int, int, int, ...). This is a broad pattern. A @Mixin on each concrete vanilla container class (ContainerWorkbench, ContainerEnchantment, ContainerRepair) may be cleaner than trying to replicate the generic ASM scan. Confirm this approach is acceptable, or whether a @Pseudo mixin on a shared supertype is preferred.
ServerConfigurationManagerAdapter uses a heuristic — it tracks the last ALOAD before loadChunk to find the new player variable. The Mixin @Redirect approach is cleaner and more reliable, but we need to confirm the exact method signature of respawnPlayer to write the @Inject/@Redirect correctly.
Client-only mixin for EntityRenderer — MixinEntityRenderer must use @Mixin with targets or be placed in a client-only source set to avoid server-side class-not-found errors.
